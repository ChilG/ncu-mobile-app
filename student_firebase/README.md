## 5. คำถามทบทวนและโจทย์ทดสอบความรู้ (ท้ายบทเรียน)

### คำถามข้อที่ 1:
**เหตุใดข้อมูลบนหน้าจอแอปพลิเคชันของเราจึงเปลี่ยนแปลงทันทีเมื่อมีการอัปเดตข้อมูลบนหน้าเว็บ Firebase Console โดยที่ไม่จำเป็นต้องดึงข้อมูลใหม่ (Pull-to-refresh)? จงอธิบายกลไกของ StreamBuilder ร่วมกับ Firestore**

**คำตอบ:**
กลไกการอัปเดตข้อมูลแบบเรียลไทม์ (Real-time update) เกิดจากการทำงานร่วมกันของ **Cloud Firestore Snapshots** และ **StreamBuilder** ของ Flutter ดังนี้:
1. **Firestore Real-time Streams (`snapshots()`):** Cloud Firestore ใช้โปรโตคอลการเชื่อมต่อแบบถาวร (Persistent Connection) เช่น WebSockets หรือ HTTP/2 เชื่อมต่อกับแอปพลิเคชันตลอดเวลา เมื่อมีการเปลี่ยนแปลงใดๆ บนฐานข้อมูล (เช่น การแก้ไขข้อมูลบน Firebase Console) เซิร์ฟเวอร์ Firestore จะทำการส่งแจ้งเตือนการเปลี่ยนแปลงข้อมูลส่วนต่าง (Data changes) มายังอุปกรณ์ฝั่งผู้ใช้อัตโนมัติในรูปแบบของ Stream ของ `QuerySnapshot`
2. **Flutter `StreamBuilder`:** ทำหน้าที่คอยตรวจสอบและฟังกิจกรรม (Listen) ที่เกิดขึ้นใน Stream นั้น เมื่อได้รับข้อมูลใหม่ (Snapshot) เข้ามา `StreamBuilder` จะสั่งให้ระบบทำการทริกเกอร์วิดเจ็ตเพื่อวาดหน้าจอใหม่ (Rebuild Widget) ในตำแหน่งเฉพาะโดยอัตโนมัติ ทำให้ผู้ใช้เห็นข้อมูลที่แก้ไขล่าสุดบนหน้าจอในทันทีโดยไม่ต้องทำการกดรีเฟรชข้อมูลเอง

### คำถามข้อที่ 2:
**ในกรณีที่อุปกรณ์เคลื่อนที่ของผู้ใช้ขาดการเชื่อมต่ออินเทอร์เน็ต (Offline) ฟีเจอร์ "Local Persistence" ของ Cloud Firestore จะมีบทบาทและจัดการกับข้อมูลการเพิ่มหรือแก้ไขในขณะนั้นอย่างไรบ้าง?**

**คำตอบ:**
ฟีเจอร์ **Local Persistence** (การเก็บข้อมูลไว้ในเครื่องแบบออฟไลน์) จะช่วยดูแลระบบผ่านกระบวนการดังนี้:
1. **Offline Caching:** Firestore SDK จะทำการแคชข้อมูลนักศึกษาที่แอปพลิเคชันเคยดึงมาแสดงผลเก็บไว้ในอุปกรณ์โดยอัตโนมัติ ทำให้แม้จะไม่มีอินเทอร์เน็ต ผู้ใช้ก็ยังสามารถเปิดดูรายชื่อนักศึกษาเดิมได้
2. **Operation Queue & Latency Compensation:** เมื่อมีการทำธุรกรรมเพิ่มหรือแก้ไขข้อมูลในขณะออฟไลน์ คำสั่งเหล่านั้นจะถูกเก็บสะสมลงในคิวงานภายในเครื่อง (Local Queue) และแอปพลิเคชันจะอัปเดตหน้าจอทันทีเสมือนว่าคำสั่งสำเร็จแล้วเพื่อไม่ให้แอปพลิเคชันเกิดอาการค้างหรือใช้งานไม่ได้
3. **Data Synchronization:** ทันทีที่อุปกรณ์กลับมาออนไลน์และเชื่อมต่ออินเทอร์เน็ตได้อีกครั้ง Firestore SDK จะประมวลผลและส่งคิวงานที่ค้างอยู่ (Local Write Queue) ขึ้นไปอัปเดตบน Cloud Firestore บนคลาวด์โดยอัตโนมัติเพื่อประสานข้อมูลให้ตรงกัน

### คำถามข้อที่ 3:
**นอกจากการส่งค่า URL สตริงของรูปภาพแล้ว หากต้องการอัปโหลดไฟล์ภาพจริงที่ถ่ายจากกล้องถ่ายรูปในเครื่องเข้าไปเก็บไว้บนระบบออนไลน์ของ Firebase จะต้องเลือกใช้บริการส่วนใดของระบบนิเวศน์ Firebase?**

**คำตอบ:**
หากต้องการอัปโหลดและจัดเก็บไฟล์ภาพดิบ (ภาพถ่ายจริง) ของผู้ใช้เข้าไปในระบบออนไลน์ จะต้องเลือกใช้บริการ **Firebase Storage** (หรือ Cloud Storage for Firebase) ซึ่งเป็นระบบจัดเก็บไฟล์สื่อ (Object Storage) ที่ออกแบบมาเพื่อรองรับไฟล์ขนาดใหญ่ เช่น รูปภาพ เสียง หรือวิดีโอ 

**กระบวนการทำงานที่ถูกต้องคือ:**
1. นำไฟล์รูปถ่ายอัพโหลดขึ้นไปเก็บที่ **Firebase Storage** ก่อน
2. เมื่ออัปโหลดสำเร็จ จะได้ลิงก์ที่อยู่ไฟล์ออนไลน์ในรูปแบบ **Download URL** กลับมา
3. จากนั้นจึงนำค่าสตริงของ **Download URL** นั้นมาบันทึกเก็บลงใน **Cloud Firestore** อีกทีหนึ่งเพื่อเป็นตัวอ้างอิงตำแหน่งของรูปภาพ
